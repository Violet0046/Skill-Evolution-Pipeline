sessions.jsonl → X 个 ProtoAnalysis
                    ↓
            1 次 LLM 调用（EvidenceAnalyzer）
                    ↓
            1 个 ExecutionAnalysis（含 N 条 evolution_suggestions）
                    ↓
            N 次 LLM 调用（SkillEvolver，每条 suggestion 一次）
                    ↓
            N 个进化产物

EvidenceAnalyzer 只调用 1 次，它同时分析所有 session 的证据，发现共性问题
evolution_suggestions 可以有 0~N 条，不是跟 session 数量一一对应
SkillEvolver 每条 suggestion 调用 1 次，产出 1 个进化产物

evidence_text.md 发给 LLM 的证据文本       可选，调试用，想看 LLM 输入了什么
execution_analysis.json	2KB	LLM 分析结果	必要，核心产物，review 时必须看
evolution_results.json	1.3KB	进化执行结果	必要，知道每条 suggestion 的成败

并发架构
两层并发模型

Layer 1: 多Skill并行 (asyncio.gather)
┌─────────────────────────────────────────────┐
│  run_pipeline_multi()                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │ skill-A  │ │ skill-B  │ │ skill-C  │    │
│  │ Stage1-7 │ │ Stage1-7 │ │ Stage1-7 │    │
│  └──────────┘ └──────────┘ └──────────┘    │
│  max_concurrent_skills=3                    │
└─────────────────────────────────────────────┘

Layer 2: 单Skill内建议并行
┌─────────────────────────────────────────────┐
│  SkillEvolver.evolve_parallel()             │
│  FIX suggestions: serial (修改原文件)        │
│  DERIVED suggestions: parallel (各自独立)    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │derived-1 │ │derived-2 │ │derived-3 │    │
│  └──────────┘ └──────────┘ └──────────┘    │
│  max_concurrent_suggestions=3               │
└─────────────────────────────────────────────┘
使用方式

# 单skill（向后兼容）
python -m skill_evolution.pipeline.runner

# 多skill并行
python -m skill_evolution.pipeline.runner --skill-names "skill-a,skill-b,skill-c"

# 或在config.yaml中配置
# skill_names: ["skill-a", "skill-b", "skill-c"]
# max_concurrent_skills: 3

# 子命令
python -m skill_evolution.pipeline.runner version
python -m skill_evolution.pipeline.runner validate
关键设计决策
决策	原因
FIX建议串行	修改原文件，后续建议依赖前序修改
DERIVED建议并行	各自输出到独立目录，互不干扰
asyncio.Semaphore	控制并发数，避免API限流
run_in_executor	LLM调用是同步的，放到线程池避免阻塞事件循环
